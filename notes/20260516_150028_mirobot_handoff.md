## MT4 실험에서 배운 점

1. 실제 로봇 이식을 목표로 하면 simplified geometry만으로는 한계가 빠르게 드러난다.
   - 집게 길이, 조인트 위치, gripper center, end-effector 방향이 조금만 달라도 학습 목표가 실제 동작과 어긋난다.
   - 따라서 다음 프로젝트는 처음부터 실제 Mirobot URDF/USD 변환 모델을 기준으로 해야 한다.

2. 단순 reach 성공률만 보면 잘못된 행동이 좋은 정책으로 선택될 수 있다.
   - 빨간 목표를 뚫고 지나가거나, 목표를 친 뒤 멈추는 행동이 reward 상으로 좋아질 수 있었다.
   - 실제 물체 집기에서는 목표 물체를 치면 물체가 이동하므로 실패로 봐야 한다.
   - 다음 task에는 목표 물체 접촉/관통/충돌을 명시적으로 벌점화해야 한다.

3. 집기 전 단계는 reach 하나가 아니라 여러 하위 문제로 나누는 것이 이해하기 쉽다.
   - 예: 정렬 -> pregrasp 위치 접근 -> gripper center 정렬 -> 최종 삽입 -> gripper close -> lift.
   - 다만 stage를 너무 많이 나누면 stage gate가 병목이 될 수 있다.
   - 다음에는 stage별 metric을 남기되, 처음부터 최종 성공까지 이어지는 최소 curriculum으로 설계한다.

4. 파란 guide marker 방식은 교육적으로 유용했다.
   - 학생들에게 “최종 목표와 중간 목표의 차이”를 시각적으로 설명하기 좋다.
   - 하지만 실제 task에서는 marker가 정답을 너무 강하게 알려주면 policy가 marker만 따라가고 물리적 의미를 잃을 수 있다.
   - Mirobot에서는 guide marker를 디버그/시각화용으로 쓰고, reward는 end-effector pose와 object pose 기준으로 정의하는 것이 좋다.

5. metric 기록과 best checkpoint 선택 기준이 중요하다.
   - success_rate만으로는 부족했다.
   - `mean_distance`, alignment, target contact penalty, overshoot, stage readiness 같은 지표가 필요했다.
   - 다음 프로젝트도 처음부터 TensorBoard scalar, checkpoint summary CSV, 실험 기록 Markdown을 자동 생성하도록 설계한다.

## 추천 작업 순서

1. Asset 정리
   - Mirobot URDF, mesh, joint limit, gripper 구조 확인
   - Isaac Sim에서 URDF import 후 USD 저장
   - joint 이름, joint axis, home pose, end-effector link, gripper finger link 확인

2. 최소 task 생성
   - 새 source package 또는 기존 project repo 안에 Mirobot direct task 추가
   - 기존 MT4 task 이름은 유지하고, Mirobot은 별도 task 이름으로 등록
   - IsaacLab core/upstream은 수정하지 않고 project repo 중심으로 관리

3. 검증 스크립트 추가
   - `scripts/train_mirobot_reach_128_1000.sh`
   - `scripts/train_mirobot_visual_16_300.sh`
   - `scripts/play_mirobot_best.sh`
   - `scripts/plot_and_select_mirobot_best.sh`
   - 기존 MT4 스크립트의 구조를 재사용하되 task 이름과 log dir을 분리

4. 1차 reward 설계
   - gripper center와 pregrasp target 거리 감소
   - end-effector approach direction alignment
   - target object와 조기 접촉 벌점
   - action/joint velocity/time penalty
   - success bonus는 처음에는 완화하고, 성공률이 생긴 뒤 엄격화

5. 교육용 기록
   - 학생용 문서에는 “왜 실제 URDF가 필요한가”를 먼저 설명
   - 실험표는 seed, env 수, iteration, reward 설정, success_rate, mean distance, contact penalty, 관찰 메모로 구성
   - 선생님 제안과 Codex 제안을 구분해서 decision log 유지
